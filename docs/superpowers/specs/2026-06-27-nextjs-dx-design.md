# Next.js Developer Experience — Design

**Date:** 2026-06-27
**Status:** Approved (sections 1–6), pending written-spec review
**Scope:** Use cases 1 (existing Next.js + Stripe shops) and 2 (greenfield
Next.js / React projects). Use cases 3 (Shopify) and 4 (WooCommerce) are
out of scope and deferred.

## Problem

The current Steelyard surface (`serveCommerce(manifest).listen(3000)`)
assumes a standalone Node HTTP server. It does not drop into a Next.js
App Router app that already owns its routing, middleware, and Stripe
wiring. The two highest-leverage user segments — existing Stripe shops
on Next.js, and greenfield Next.js builds — currently see Steelyard as
an additional server they have to host, not a library they can adopt
incrementally.

The goal of this release is to make adoption feel like adding any other
Next.js integration: one CLI command, three prompts, working agent
surfaces in under a minute. The CLI is treated as a first-class part of
the product, not a developer convenience.

## Goals

1. A user with an existing Next.js + Stripe app can add agent-discovery
   surfaces (`/.well-known/commerce.json`, `/mcp`, `/acp/feed`,
   `/.well-known/ucp`) in under 60 seconds, without modifying their
   existing checkout.
2. A greenfield user can scaffold the same surfaces against a sample
   catalog with the same command.
3. Either user can upgrade from discovery-only (tier A) to agent
   checkout (tier B) with a single follow-up command.
4. The CLI experience is delightful — ASCII banner, smooth animations,
   thoughtful prompts, recoverable errors.
5. A dev-mode inspector at `/__steelyard` lets users see what agents see
   without running an agent.

## Non-Goals

- Shopify and WooCommerce adapters. Those are cloud-side / PHP problems
  tracked separately.
- Replacing the user's human web checkout. Tier C ("Steelyard owns
  checkout for humans too") was explicitly ruled out.
- Pages Router demo project. The `@steelyard/next` adapter ships a
  Pages helper, but `examples/nextjs` is App Router only.
- Vercel "Deploy" button, webhook-driven manifest refresh, telemetry,
  account/login. All deferred.
- Schema extensions for `trial_days`, arbitrary recurring intervals,
  tiered pricing, setup fees, tax behavior. Tracked in `TODO.md` for a
  follow-up GOAL.

## Tiers

| Tier | What's mounted | Money moves? | Stripe key required? |
|------|----------------|--------------|----------------------|
| A | Discovery surfaces only | No | No |
| B | Discovery + agent checkout via existing Stripe | Yes (agent purchases) | Yes |

Tier A is the default. `steelyard enable checkout` upgrades to B. Tier C
(Steelyard owns human checkout) is rejected by design.

## Architecture

### Packages

- **`@steelyard/next`** (new) — Next.js adapter. Exports:
  - `toNextHandler(nodeHandler)` — wraps existing Node-style handlers
    into App Router `Request → Response` handlers.
  - `toNextApiHandler(nodeHandler)` — Pages Router equivalent.
  - `createCommerceRoutes(manifest, opts)` — returns the bundle of
    handlers used by codegen.
  - `createInspectorRoute()` — the dev inspector, dev-only at
    `app/(steelyard)/__steelyard/page.tsx`.
- **`@steelyard/cli`** (exists) — gains `steelyard init`, `steelyard
  enable checkout`, and underlying detection/codegen modules.
- **`examples/nextjs`** (new) — the demo project. Generated by running
  `steelyard init` against a stock `create-next-app` output, then
  committed. Doubles as a CLI integration test and the reference
  implementation users will end up with.

### File layout after `steelyard init`

```
app/
  .well-known/commerce.json/route.ts    # 3 lines — boilerplate, user-owned
  mcp/route.ts                          # 3 lines
  acp/feed/route.ts                     # 3 lines
  api/ucp/[...path]/route.ts            # 3 lines
  (steelyard)/__steelyard/page.tsx      # dev inspector, opt-in at init
commerce.ts                             # the only file users edit after init
.env.local                              # STRIPE_SECRET_KEY (tier B only)
```

The contract: **`commerce.ts` is the only file users touch after
init.** Route files are owned by the user (in their repo, edited by
their formatter) but should never need to change.

### Manifest authoring (`commerce.ts`)

```ts
import { defineCommerce } from "steelyard";

export default defineCommerce({
  identity: { name: "Acme Coffee", domain: "acme.coffee", currencies: ["USD"] },
  offers: [
    {
      id: "single",
      title: "Single Espresso",
      availability: "in_stock",
      pricing: [{ kind: "one_time", amount: 300, currency: "USD" }],
      stripe: { priceId: "price_1ABC..." }  // present only after `enable checkout`
    }
  ]
});
```

Behaviors:

- **Async/dynamic manifests supported.** `export default async () =>
  defineCommerce(...)` works for users pulling offers from a DB.
- **HMR-aware.** In `next dev`, editing `commerce.ts` re-renders
  `/.well-known/commerce.json` on next request without a server restart.
  Implementation: handlers dynamic-import the manifest so Next's module
  graph invalidation handles it.
- **Single source of truth.** Route files import from a configurable
  path (default `./commerce`).
- **Typed end-to-end.** `stripe.priceId` added to the Offer type via a
  `declare module` augmentation in `@steelyard/next`.
- **Path is configurable.** Default is repo root next to `next.config.ts`;
  CLI accepts `--manifest path/to/file.ts` and always prompts (Enter to
  accept default), per create-next-app convention.

## CLI as a first-class experience

The CLI is not a developer convenience — it is the product surface most
users will judge Steelyard by in the first minute. It must feel as
considered as `create-next-app` or `vercel deploy`.

### Craft requirements

- **ASCII banner** at the top of `steelyard init`. Branded, single
  color, ≤6 lines, drawn once at start. Suppressed in non-TTY
  environments (CI, redirected output) and by `--no-banner`.
- **Smooth animations.** Spinners use `ora` (or equivalent) with
  Unicode dots, never spammy. Each long-running step has a verb in the
  active form ("Fetching Stripe catalog…") and resolves to a green
  check with elapsed time ("✔ Fetched 12 prices · 480ms").
- **Color discipline.** Cyan for actions, green for success, yellow for
  warnings, red for errors. Dim gray for incidental info. Respects
  `NO_COLOR` and non-TTY.
- **Typography.** Hierarchy via blank lines and indentation, not via
  ASCII boxes or heavy separators. Bold for section headers, dim for
  meta. Prompts use the create-next-app shape (`?` cyan, question,
  faint default, then input).
- **Micro-interactions.** When a file is written, the path is printed
  briefly with a dim "writing…" then resolved to "✔ wrote
  app/mcp/route.ts". When a prompt is answered, the chosen value
  remains on screen above the next prompt for context.
- **Closing screen.** "Next:" block with 2–3 commands, each on its own
  line, copy-pasteable. Never more than 5 lines total.
- **Recoverable errors.** Any write that fails rolls back the entire
  init (transactional). Errors print the failing path, the cause, and
  the recovery action ("Run `steelyard init` again to retry").
- **No telemetry, no account, no login.** It is a codegen tool.

### Detection rules

| Signal | Source | If found | If not |
|---|---|---|---|
| Framework | `package.json` `dependencies.next`, presence of `app/` vs `pages/` | Use detected router | Fallback to generic Node target |
| TS vs JS | `tsconfig.json` presence | `.ts` route files | `.js` route files |
| Package manager | lockfile (`pnpm-lock.yaml`, `yarn.lock`, `package-lock.json`, `bun.lock`) | Use detected | Prompt |
| Stripe present | `dependencies.stripe`, env var scan for `STRIPE_*` | Offer catalog import | Offer test-mode wiring with sample catalog |
| Existing routes | scan target paths under `app/` | Ask before overwriting, diff shown | Write |

### Stripe import: lossy mapping handling

The manifest schema represents one-time, recurring (month/year),
usage_based, contact_sales, and multi-currency (via repeated `pricing`
entries). What Stripe has that doesn't map cleanly:

- `recurring.trial_days` — no schema field
- Arbitrary intervals (day, week, `interval_count`) — schema is
  month/year only
- Tiered / graduated / volume pricing — `usage_based` is single-rate
- Subscription setup fees — not represented
- Tax behavior (inclusive/exclusive) — no field
- Archived products — skipped silently

For this release: import cleanly-mappable prices, **warning-summary the
rest** at the end of init with the price ID, the reason, and a link to
the tracking issue. Per-offer prompting is rejected (kills the 30s
target); silent skipping is rejected (erodes trust). Schema extensions
to remove these limitations are in `TODO.md` for a follow-up GOAL.

### `steelyard enable checkout`

The tier-A → tier-B upgrade, one command:

```
$ npx steelyard enable checkout

  ⚠ This will start accepting agent purchases against your Stripe account.

? Which Stripe key?  (using STRIPE_SECRET_KEY from .env.local) ›
  ⠋ Verifying key…
  ✔ Connected to acct_… (test mode)
  ⠋ Matching manifest offers to Stripe Prices…
  ✔ Matched 12/12 offers

  Updating files…
  ✔ commerce.ts (added stripe.priceId to 12 offers)
  ✔ app/api/ucp/[...path]/route.ts (enabled checkout)
  ✔ app/acp/feed/route.ts (enabled checkout)

  Next:
    Test:  curl localhost:3000/.well-known/ucp
```

### Idempotency

- Re-running `init` updates route files in place. `commerce.ts` is
  diffed (offers added / removed / changed in Stripe) and merged with
  user confirmation. The user's hand edits to `commerce.ts` are
  preserved.
- Re-running `enable checkout` is a no-op if already enabled.

## Dev inspector (`/__steelyard`)

A single Next.js page at `app/(steelyard)/__steelyard/page.tsx`, written
by the CLI, guarded by `if (process.env.NODE_ENV !== "production")` in
the handler. Zero bytes shipped to prod bundles. User-owned: visible in
their repo, gitignorable, deletable.

Panels:

1. **Live manifest view.** Pretty-printed `/.well-known/commerce.json`
   with spec-compliance badges per surface ("mounted at `/mcp`",
   "spec compliant", "last regenerated 2s ago").
2. **Surface ping table.** Pings each mounted surface, shows raw
   response and round-trip latency. Click to expand.
3. **Stripe drift panel** (tier B only). For each offer with a
   `stripe.priceId`, fetches the live Stripe Price and flags mismatches
   (currency, amount, archived). One-click "sync manifest from Stripe".
4. **Agent traffic tail.** Last 20 requests to any Steelyard route with
   headers, body, response, matched handler. The "why isn't ChatGPT
   seeing my catalog" debugger.
5. **Try-it shelf.** Buttons that call `tools/list` on `/mcp`,
   `GET /acp/feed`, `GET /.well-known/ucp` from the browser with sane
   defaults so users can see what an agent sees without an agent.

## `examples/nextjs/` demo project

A real, deployable Next.js 15 App Router app under `examples/nextjs/`,
runnable with `pnpm --filter @steelyard/example-nextjs dev`.

- **Story:** "Acme Coffee" — reuses the three SKUs from `coffee-shop`
  (single, double, cappuccino) so existing test infra carries over, but
  presented as a real shop with a Tailwind landing page, product grid,
  and human Stripe Checkout.
- **Two checkout paths in one app:**
  - **Human checkout:** vanilla Stripe Checkout via `@stripe/stripe-js`
    and a `POST /api/checkout` route. Untouched by Steelyard.
  - **Agent surfaces:** the four Steelyard routes, tier B wired to a
    Stripe test key.
- **Tier toggle:** `STEELYARD_TIER=a|b` switches between
  discovery-only and checkout-enabled. Same code, no branching,
  demonstrates the upgrade path directly.
- **Built via the CLI, not by hand.** Generated by running `steelyard
  init` against a stock `create-next-app` output, then committed. Two
  benefits: (1) it's an integration test of the CLI; (2) it is literally
  the file tree users will end up with.
- **`private-docs/loom-script-nextjs-demo.md`** — 2-minute recording
  script: init → dev inspector → ChatGPT via MCP fetches catalog →
  agent purchase.
- **CI:** `pnpm --filter @steelyard/example-nextjs build` and a smoke
  test that boots the app and hits the four well-known surfaces,
  asserting spec compliance. Shape matches the coffee-shop smoke tests.

## Release surface

Following the project's release-doc rules (CLAUDE.md):

- `CHANGELOG.md` gets a new section with `Added`, `Changed`, and
  `Fixed` entries referencing GOAL.md criterion IDs.
- `docs/releases.md` gets a narrative entry mirroring the GitHub
  Release body.
- Both files updated on the integration commit, not at tag time.

## Risks and open questions

- **HMR semantics.** Dynamic-importing `commerce.ts` on each request in
  dev works but adds latency. Confirm acceptable (likely <10ms).
- **Route group naming.** `app/(steelyard)/` claims a route group
  name; collision with user's own naming is unlikely but possible.
  Detect and prompt.
- **CLI banner in CI.** Banner suppression via TTY detection should
  cover most cases; confirm against GitHub Actions and Vercel build
  output.
- **Tier B + dynamic manifest.** A DB-backed manifest interacts oddly
  with `stripe.priceId` matching at init time. Init only runs once;
  subsequent DB changes won't be matched. Document this as a known
  limitation; revisit in a follow-up.
