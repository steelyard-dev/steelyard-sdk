# x402 Paid HTTP Resources

x402 is an HTTP payment overlay for paid resources. A server returns
`402 Payment Required` with a `PAYMENT-REQUIRED` challenge, the buyer wallet
evaluates local policy, signs an x402 payment payload through a configured
signer, retries with `PAYMENT-SIGNATURE`, and receives settlement metadata in
`PAYMENT-RESPONSE`.

Use x402 when the unit being sold is a request to an HTTP resource: an API call,
dataset row, model inference, report, or similar metered endpoint.

Use ACP or UCP checkout when the unit being sold is commerce: catalog discovery,
cart or offer checkout, PSP capture, fulfillment, and receipts. x402 does not
require a Steelyard commerce manifest and does not replace ACP/UCP checkout.

## Steelyard Model

Buyer side:

- `x402Payments(...)` creates an agent-native payment instrument.
- `x402Fetch(wallet, opts)` wraps `fetch` and handles the x402 retry loop.
- Wallet policy runs before any x402 payload is signed.
- Signers are bring-your-own adapters; the public API does not accept raw
  private key strings.

Server side:

- `x402Paywall(...)` protects explicit HTTP routes.
- `exactUsdc(...)` declares a USDC `exact` payment requirement.
- Verification and settlement go through an `X402FacilitatorClient`.
- Steelyard does not operate a facilitator.

## Safety Defaults

The first implementation is testnet-first. Mainnet signer configuration must
opt in explicitly with `allowMainnet: true`. The public
`https://x402.org/facilitator` endpoint is useful for demos and experiments,
not a production default.

Default examples use Base Sepolia (`eip155:84532`) and mock facilitators. Tests
do not require funded wallets or live chain access.

## Relationship To Mandates

For Steelyard, the signed x402 payment payload is represented as a short-lived
`PaymentMandate`. The wallet creates it only after the `PAYMENT-REQUIRED`
challenge has been parsed, a supported requirement has been selected, and local
policy has allowed the intent.
